# 言娘 憲法

## ビジョン

**最終目標**: 現実の場で、日本語を介さず自由に中国語で対話できるようになること。

中国語学習における最大の障壁は「日本語で考えてから中国語に訳す」習慣である。
このアプリは、**中国語と個人の体験・感情・感覚を直接結びつける**ことで、
その習慣を根本から解消することを目指す。

理想の状態は「似た状況に遭遇したとき、翻訳なしに中国語表現がフッと浮かぶ」こと。

### 戦略の3本柱

1. **言語化・記録** — 過去・現在の体験や感情を中国語で書き留める。
   日本語訳を主役にせず、中国語と体験を直結させる記録を積み上げる。

2. **リマインド → 永久記憶** — アプリが記録をもとにリマインドし、
   体験と表現の結びつきを時間をかけて強化する（間隔反復）。

3. **関連コンテンツ発見** — 新たに覚えた語彙・フレーズに関連する動画・画像・音声を
   ネットから自動収集し、視聴覚体験とともに定着させる。

## 設計原則

### 1. 日本語介在の排除

UI・学習フロー全体を通して、日本語訳を主たる意味説明の手段としてはならない（MUST NOT）。
意味の説明は中国語・状況・画像・音声で行わなければならない（MUST）。
日本語は補助的な参照としてのみ許容し、学習の主軸に置いてはならない（MUST NOT）。

**根拠**: 日本語介在が習慣化すると、リスニング・スピーキングともに
一旦日本語を経由する回路が固定され、自然な中国語運用が妨げられる。

### 2. 個人文脈との結びつき

学習項目（語彙・フレーズ・表現）は必ず個人の体験・感情・感覚と紐づけて記録しなければならない（MUST）。
「どんな状況で・何を感じたときに・この表現を使うか」のコンテキストを保存する設計にしなければならない（MUST）。
汎用的な例文のみに依存した学習フローを提供してはならない（MUST NOT）。

**根拠**: 人間の記憶は感情・状況と結びついたとき最も定着する。
個人文脈のないフラッシュカード型学習は短期記憶にとどまりやすい。

### 3. 視聴覚体験の重視

語彙・フレーズの定着には、テキストだけでなく動画・画像・音声コンテンツを積極的に活用しなければならない（MUST）。
関連コンテンツの収集は自動化し、ユーザーが手動で検索する手間を最小化しなければならない（MUST）。
コンテンツは中国語ネイティブ向けの素材を優先しなければならない（MUST）。

**根拠**: 視聴覚記憶はテキスト記憶より強固であり、
ネイティブ向け素材への接触は自然な語感を養う最短経路である。

### 4. シンプルさ（過剰設計の禁止）

1 つの機能のために 3 つ以上の新しい抽象レイヤーを追加してはならない（MUST NOT）。
ヘルパー・ユーティリティは 2 箇所以上で使われる場合にのみ作成する（MUST）。
要求された機能の範囲を超えた投機的な実装を行ってはならない（MUST NOT）。
複雑さを追加する場合は Complexity Tracking で明示的に正当化しなければならない（MUST）。

**根拠**: 個人プロジェクトが過剰設計で止まることを防ぐ。
動くシンプルなものが、完璧を目指して止まるものより常に価値がある。

## 技術スタック

以下のスタックを標準とする。代替ライブラリの追加は明示的な合意なしに行ってはならない（MUST NOT）。

| レイヤー               | 技術                              | 備考 |
| ---------------------- | --------------------------------- | ---- |
| プラットフォーム       | Android                           | —    |
| 言語                   | Kotlin                            | —    |
| UI                     | Jetpack Compose                   | —    |
| アーキテクチャ         | MVVM（ViewModel + StateFlow）     | —    |
| ローカル DB            | Room                              | SQLite ORM |
| HTTP クライアント      | Ktor Client                       | coroutines 親和性 |
| HTML 解析（OGP取得）   | Jsoup                             | og:title / og:image |
| DI                     | Koin                              | —    |
| 非同期                 | Kotlin Coroutines + Flow          | —    |
| ナビゲーション         | Compose Navigation                | —    |
| バックグラウンド処理   | WorkManager                       | 「依頼→通知」フロー用（将来） |
| AI（コンテンツ収集）   | Anthropic API（Ktor 経由で直接呼び出し） | 公式 Android SDK なし |

**型安全性**: Kotlin の `!!`（non-null assertion）の使用を禁止し、`Result` 型または明示的なエラー処理を使用する（MUST）。
Room の DAO 入出力は data class で型定義する（MUST）。
Ktor のリクエスト・レスポンスは kotlinx.serialization で定義された型付き構造体を使用する（MUST）。

## 開発ワークフロー

- **コードチェック**: PR は `./gradlew lint` をパスしなければならない（MUST）。
- **ビルド確認**: `./gradlew assembleDebug` が成功することを確認する（MUST）。
- **ブランチ**: `main` への直接プッシュを禁止する（MUST NOT）。機能は連番ブランチ（例: `001-feature-name`）で開発する。
- **コミット粒度**: タスク単位でコミットし、複数機能を 1 コミットにまとめてはならない（MUST NOT）。

## ガバナンス

この憲法はプロジェクトの他すべての慣習・ガイドラインより優先される。
すべての機能設計（仕様・計画・タスク）は憲法チェックで本文書との整合性を確認しなければならない（MUST）。

**改訂手順**:

1. 改訂の必要性と理由を記録する。
2. 影響を受けるテンプレート・仕様・タスクを特定する。
3. セマンティックバージョニングに従ってバージョンをインクリメントする。
4. `LAST_AMENDED_DATE` を更新し、変更影響レポートを更新する。

**バージョニングポリシー**:

- MAJOR: 原則の削除または後方互換性のない再定義。
- MINOR: 原則・セクションの追加または実質的な拡張。
- PATCH: 表現の明確化・誤字修正など意味を変えない修正。

**コンプライアンスレビュー**: `/speckit.plan` 実行時に憲法チェックを必須ゲートとして実施する。
